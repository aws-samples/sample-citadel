# Citadel - Deployment Guide

Complete guide for deploying the Citadel application with AgentCore integration support.

## Quick Start

```bash
# 1. Configure environment
cp backend/.env.example backend/.env
# Edit backend/.env with your AWS account and admin credentials

# 2. Deploy everything
export AWS_PROFILE="your-aws-profile"
./deploy.sh --profile your-aws-profile
```

That's it! The Gateway ID is automatically configured from the Services Stack.

## Prerequisites

- **Node.js** 24+ installed
- **Python** 3.14+ installed
- **AWS CLI** configured with credentials
- **AWS CDK** CLI installed: `npm install -g aws-cdk`
- **Container runtime** — Docker or Finch (for Lambda bundling, see below)
- **AWS Account** with appropriate permissions

### Container Runtime (Finch instead of Docker Desktop)

Docker Desktop requires a commercial license for organisations with 250+ employees or $10M+ revenue. [Finch](https://github.com/runfinch/finch) is an open source alternative that works with AWS CDK.

Install Finch via Homebrew:

```bash
brew install --cask finch
```

Initialise and start the VM (first time only):

```bash
finch vm init
finch vm start
```

Tell CDK to use Finch instead of Docker by setting the `CDK_DOCKER` environment variable:

```bash
export CDK_DOCKER=finch
```

Add this to your shell profile (`~/.zshrc` or `~/.bashrc`) to make it permanent:

```bash
echo 'export CDK_DOCKER=finch' >> ~/.zshrc
```

All CDK commands (`cdk synth`, `cdk deploy`, etc.) will then use Finch for container operations.

## Configuration

### 1. Environment Variables

Create and configure the environment file:

```bash
cp backend/.env.example backend/.env
```

Edit `backend/.env`:

```bash
# Required
ENVIRONMENT=dev
CDK_DEFAULT_ACCOUNT=123456789012  # Your AWS account ID
CDK_DEFAULT_REGION=ap-southeast-2  # Your preferred region

# Admin User (required for first deployment)
ADMIN_EMAIL=admin@example.com
ADMIN_FIRST_NAME=Admin
ADMIN_LAST_NAME=User
```

**Note:** The admin password is auto-generated by AWS Secrets Manager during deployment. You do not need to set it manually. See [Post-Deployment](#post-deployment) for how to retrieve it.

### 2. AWS Profile

Set your AWS profile:

```bash
export AWS_PROFILE="your-profile-name"
```

Or use the `--profile` flag with deployment commands.

## Deployment Steps

### Step 1: First Deployment

Deploy all stacks:

```bash
export AWS_PROFILE="your-profile-name"
./deploy.sh --profile your-profile-name
```

**What gets deployed:**
- ✅ Backend Stack: AppSync API, Cognito, DynamoDB, Lambda functions
- ✅ Services Stack: Bedrock agents, AgentCore Gateway, Knowledge Base
- ✅ Arbiter Stack: Agent orchestration system
- ✅ Frontend Stack: React app on S3/CloudFront
- ✅ Gateway Stack: AgentCore Gateway configuration

**Duration:** 15-20 minutes

The Gateway ID is automatically imported from the Services Stack and passed to the Integration Resolver Lambda function. No manual configuration needed!

## Post-Deployment

### 1. Get Application URL

Find the CloudFront URL in deployment outputs:

```bash
aws cloudformation describe-stacks \
  --stack-name citadel-frontend-dev \
  --query 'Stacks[0].Outputs[?OutputKey==`CloudFrontUrl`].OutputValue' \
  --output text
```

### 2. Retrieve Admin Password

The admin password is auto-generated by AWS Secrets Manager. Retrieve it with:

```bash
aws secretsmanager get-secret-value \
  --secret-id citadel/admin-password-dev \
  --query SecretString \
  --output text | python3 -c "import sys,json; print(json.load(sys.stdin)['password'])"
```

Or get the secret ARN from the stack outputs:

```bash
aws cloudformation describe-stacks \
  --stack-name citadel-backend-dev \
  --query 'Stacks[0].Outputs[?OutputKey==`AdminPasswordSecretArn`].OutputValue' \
  --output text
```

### 3. Access Application

Navigate to the CloudFront URL and login with:
- Email: Your `ADMIN_EMAIL`
- Password: The auto-generated password from Secrets Manager (see above)
- On first login, you will be prompted to change your password

### 3. Verify Integrations

1. Navigate to Integrations page
2. Verify you see all integration types:
   - **SaaS Connectors**: Confluence, Jira, ServiceNow, Slack, Microsoft, PagerDuty, Zendesk
   - **AWS Lambda**: Execute custom business logic
   - **AWS Services (Smithy)**: Direct AWS service access (DynamoDB, S3, Lambda, SQS, SNS)
   - **MCP Server**: External MCP-compatible servers

See [INTEGRATIONS.md](INTEGRATIONS.md) for detailed integration setup.

## Troubleshooting

### Missing Environment Variables

**Error**: "Missing required environment variables"

**Solution**: Ensure `backend/.env` exists and contains all required variables:
```bash
cp backend/.env.example backend/.env
# Edit and configure all required fields
```

### AWS Credentials Not Found

**Error**: "Unable to locate credentials"

**Solution**:
```bash
# Configure AWS CLI
aws configure --profile your-profile-name

# Or set environment variables
export AWS_ACCESS_KEY_ID=your-key
export AWS_SECRET_ACCESS_KEY=your-secret
export AWS_DEFAULT_REGION=ap-southeast-2
```

### Admin User Creation Failed

**Error**: "Admin user creation failed"

**Solution**: Check that:
- `ADMIN_EMAIL` is a valid email (set in `.env` or via `--admin-email` flag)
- The Secrets Manager secret `citadel/admin-password-{env}` was created successfully
- User doesn't already exist in Cognito
- The seed Lambda has permission to read from Secrets Manager

### Gateway Configuration Issues

**Error**: "AgentCore Gateway not configured"

**Solution**:
1. Verify Services Stack deployed successfully:
   ```bash
   aws cloudformation describe-stacks --stack-name citadel-services-dev
   ```
2. Check Gateway ID is exported from Services Stack
3. Redeploy backend stack to pick up the Gateway ID:
   ```bash
   ./deploy.sh --backend-only --profile your-profile
   ```

### Build Failures

**Frontend build failed**:
```bash
cd frontend
rm -rf node_modules package-lock.json
npm install
npm run build
```

**Backend build failed**:
```bash
cd backend
rm -rf node_modules package-lock.json
npm install
npm run build
npm run build:lambda
```

### CDK Bootstrap Required

**Error**: "This stack uses assets, so the toolkit stack must be deployed"

**Solution**:
```bash
cdk bootstrap aws://ACCOUNT-ID/REGION --profile your-profile
```

### Integration Creation Fails

**Error**: "Failed to create gateway target"

**Solution**:
1. Verify Services Stack deployed successfully and exported Gateway ID
2. Check Lambda function has permissions to call AgentCore APIs
3. Review CloudWatch logs:
   ```bash
   aws logs tail /aws/lambda/citadel-backend-IntegrationResolverFunction --follow
   ```
4. Redeploy backend stack to ensure Gateway ID is imported:
   ```bash
   ./deploy.sh --backend-only --profile your-profile
   ```

```bash
  --gateway-identifier your-gateway-id \
```

```bash
  --query 'Stacks[0].Outputs[?OutputKey==`GraphQLApiUrl`].OutputValue' \
  --output text)

echo "GraphQL API: $API_URL"
```

## Clean Up

To remove all deployed resources:

```bash
export ENVIRONMENT=dev
export AWS_PROFILE="your-profile"

# Delete stacks in reverse order
aws cloudformation delete-stack --stack-name citadel-frontend-${ENVIRONMENT}
aws cloudformation delete-stack --stack-name citadel-gateway-${ENVIRONMENT}
aws cloudformation delete-stack --stack-name citadel-arbiter-${ENVIRONMENT}
aws cloudformation delete-stack --stack-name citadel-services-${ENVIRONMENT}
aws cloudformation delete-stack --stack-name citadel-backend-${ENVIRONMENT}

# Clean local artifacts
./clean.sh
```

**Warning**: This will permanently delete all data including:
- DynamoDB tables and data
- S3 buckets and files
- Cognito users
- All configurations
- Integration credentials

## Security Best Practices

1. **Never commit `.env` files** - They contain sensitive configuration
2. **Admin password is auto-generated** - Retrieved from Secrets Manager after deployment, never stored in code
3. **Temporary password** - Admin must change password on first login
4. **Use IAM roles** - For production, use IAM roles instead of access keys
5. **Enable MFA** - For admin accounts in production
6. **Review permissions** - Regularly audit IAM policies
7. **Encrypt data** - All data is encrypted at rest and in transit
8. **Monitor logs** - Review CloudWatch logs for suspicious activity
9. **Use Secrets Manager** - Store sensitive data in AWS Secrets Manager
10. **Limit access** - Use least privilege principle for IAM policies
